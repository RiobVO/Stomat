## Контекст

Живой тест после полировки-2 дал пять находок. Три — решения пользователя,
меняющие принятые ранее конвенции (зафиксировано: пользователь в праве
пересмотреть собственные решения по живому опыту); две — чистые баги.

## А. Баг: «на завтра» теряет дату (book↔other дыра)

Живой NLU классифицирует голое «на завтра» как `intent=other,
date_ref=tomorrow` (проверено прямым вызовом экстрактора). Бэкстоп
booking_like в `_route_intent` покрывает только `question`+date_ref —
`other`+date_ref проваливается мимо, дата выбрасывается, контекст-правило
переспрашивает день.

Решение: бэкстоп расширяется — текст с привязкой ко времени
(`date_ref`/`time_ref`) при интенте `question` ИЛИ `other` трактуется как
про запись (та же философия, что book↔question). Дыра старая (до П-1 дата
выбрасывалась с «не понял»+меню), не регрессия полировки-2.

## Б. Пересмотр контекст-правила П-1 (решение пользователя)

Мак-правило «любой текст посреди записи = вопрос о наличии → выбор дня»
живым тестом дало: мусор («уыкп\») на шаге УСЛУГИ прыгает на выбор дня
с дефолтным «осмотром»; «ПРИВЕТ» в зависшем booking_collect получает
«на какой день вам удобно?»; текстом из этого не выйти.

Решение: контекст-правило сужается. Посреди сценария вопрос о наличии —
только при словарном маркере (`mentions_availability`) или дате/времени
в извлечении (покрывается А). Прочий непонятый текст (question/other без
слотов) посреди сценария → повтор ТЕКУЩЕГО шага через машинерию сбоев
(`nlu_failures`: 1-й — повтор шага, 2-й — «не понял» + кнопка
«👤 Позвать администратора», как в фоллбэках полировки-2). Вне сценария
поведение не меняется (меню/копилка/FAQ как есть).

## В. Телефон пациента: хранить шифрованным (пересмотр security-решения)

Пользователь решил: владельцу в календаре нужен номер пациента. Сейчас
открытый номер не хранится нигде (SHA-256 на границе очереди, d5cd3c9).
Решение — паритет с именем: номер хранится AES-256-GCM (тот же
NAVBAT_ENC_KEY), хэш остаётся ключом поиска.

- Миграция 0018: `patient.phone_encrypted text NULL`.
- Граница очереди: payload несёт `phone_hash` + `phone_encrypted`
  (шифруется там же, где хэшируется; открытый номер в payload
  по-прежнему не кладётся).
- `create_patient_with_hash` принимает шифртекст; demo-путь
  `create_patient(raw)` шифрует сам.
- Ротация ключа (`rotate_key`) получает шестую AES-колонку.
- PRIVACY.md разд. 7/9: номер хранится шифрованным, уходит в Google
  Calendar (решение владельца продукта).
- /forget обнуляет phone_encrypted вместе с остальным.

## Г. Событие Google Calendar: русская услуга + имя + телефон

Сейчас summary = `whitening (Navbat)`. Решение пользователя — владелец
живёт в календаре, событие должно отвечать «кто и зачем»:

- summary: `{label_ru услуги} — {имя пациента} (Navbat)`;
- description: `Телефон: {номер}` (расшифрованный из patient), плюс
  существующие поля. Имя/номер могут отсутствовать (gcal_import,
  пациент без записи) — строки пропускаются.
- ПД (имя, номер) уходит в Google-аккаунт клиники — осознанный трейдофф,
  фиксируется в PRIVACY.md (разд. «внешние сервисы»).
- Перепривязка: изменение шаблона не трогает уже выгруженные события
  задним числом (бот перезапишет при следующем diff-экспорте этого
  диапазона — допустимо).

## Д. Выбор даты: возврат сетки с маркерами (вариант B, решение пользователя)

Список доступных дней (П-5б, мак) пользователь забраковал живым тыком на
Windows: «нужно листать, а ожидал сетку с точками». Возвращается вариант B
из Windows-дизайна (.superpowers/spec-backup-…, блок 5):

- Месячная inline-сетка: заголовок «Июнь 2026», ряды пн–вс, навигация
  «◀ май | июнь ▶» (editMessageText, копий в чате нет);
- свободный день — «•15» → day-view слотов дня (существующий П-5);
- день без слотов / прошлое — «15» → noop + toast answerCallbackQuery;
- горизонт 90 дней (как у списка); callback'и сырые `cal:` (как сейчас);
- legacy-callback'и старых сообщений (`cal:nav:ISO`, noop/none)
  деградируют штатно: открывают сетку месяца этой даты.
- Кнопки «Сегодня/Завтра/Послезавтра/📅 Выбрать дату» на шаге дня —
  не меняются; меняется только то, что за «📅 Выбрать дату».

## Е. Фикс OAuth-гонки (уже в рабочем дереве, закоммитить с тестом)
