# Кнопочный вход (button-first, LLM-fallback) — дизайн

Дата: 2026-06-06. Статус: утверждён пользователем (устно, в сессии).

## Проблема

Бот стартует «вслепую»: `/start` уходит в LLM как обычный текст, пациент получает
«я виртуальный администратор, напишите словами» без единой кнопки. Главного меню
нет — пациент обязан догадаться, что писать. При этом внутри сценария бот уже
кнопочный (услуги, даты, слоты, подтверждение отмены — inline; телефон — только
`request_contact`).

## Решение (принятые развилки)

1. **Кнопки-first, LLM-fallback.** Главное меню + кнопки на каждом шаге.
   Свободный текст на любом шаге по-прежнему уходит в NLU (LLM) и ведёт по тому
   же FSM — главная фишка продукта (понимает узбекский/русский/суржик)
   сохраняется, но исчезает из happy path.
2. **Постоянная reply-клавиатура** (`ReplyKeyboardMarkup`, persistent) для
   главного меню; шаги сценария остаются inline-кнопками как сейчас.
3. **Явный выбор языка при первом /start** («O'zbekcha» / «Русский», inline),
   выбор хранится в `conversation.context["lang"]`; кнопка 🌐 меняет позже.
   LLM-детекция языка по свободному тексту остаётся (BRIEF: «определение языка
   на сообщение»).

## UX-поток

```
/start (первый раз)
  → «Tilni tanlang / Выберите язык» [O'zbekcha] [Русский]   (inline)
  → выбор → приветствие-дисклеймер (P0 BRIEF) + reply-клавиатура:
      [📅 Записаться]
      [🔄 Перенести] [❌ Отменить]
      [💰 Цены]      [🌐 Til/Язык]
/start повторный → приветствие + меню, без выбора языка.
```

Маппинг меню на существующие пути FSM:

| Кнопка | Действие |
|---|---|
| 📅 Записаться | `_advance_booking` с чистым контекстом → спросит услугу (inline, есть) |
| 🔄 Перенести | `_start_reschedule` без extraction → спросит дату (есть) |
| ❌ Отменить | `_start_cancel` (есть) |
| 💰 Цены | новый ответ: весь прайс из таблицы `service` одним сообщением |
| 🌐 Til/Язык | экран выбора языка; после смены меню перерисуется со следующим menu-сообщением (см. краевые случаи) |

Нажатие reply-кнопки приходит как ТЕКСТ с label'ом — FSM перехватывает его
точным матчем по словарю label'ов (оба языка сразу, независимо от текущего
`lang`) ДО вызова NLU. Ноль токенов на happy path.

## Изменения по коду

Меняем: `dialog/fsm.py`, `dialog/replies.py`, `telegram/worker.py`,
`telegram/api.py`, `demo.py`, тесты.
Не трогаем: scheduling, calendar, reminders, NLU, миграции (язык уже живёт в
`conversation.context["lang"]`, БД не меняется).

- **`replies.py`**: в `Reply` поле `menu_keyboard: bool` (рендер постоянной
  reply-клавиатуры); новые шаблоны: `choose_lang`, label'ы меню
  (`btn_menu_book`, `btn_menu_resched`, `btn_menu_cancel`, `btn_menu_prices`,
  `btn_menu_lang`), `price_list`-строки.
- **`fsm.py`**: в `_process_text` ПЕРЕД NLU — перехват `/start` и label'ов меню.
  Перехват стоит и до PII-эвристик (`awaiting_name`/`awaiting_phone`):
  «❌ Отменить» на шаге имени корректно прерывает, имя в LLM не уходит.
  Нажатие меню посреди сценария = явная смена намерения: контекст чистится,
  висящий hold (`appointment_id`) отменяется через `_sched.cancel`,
  стартует новое действие.
- **`worker.py` / `api.py`**: рендер `ReplyKeyboardMarkup` (resize_keyboard,
  persistent). Клавиатура шлётся при /start, при смене языка и после шага
  контакта: `request_contact` затирает меню, поэтому вместо текущего
  `remove_keyboard` после подтверждения записи восстанавливаем меню
  (сознательное изменение поведения, тесты обновляются).
- **`demo.py`**: консоль рендерит меню нумерованным списком (как inline сейчас).

## Краевые случаи

- Пациент со старой клавиатурой / без /start (форвард, диплинк) → свободный
  текст → LLM-путь как сейчас; язык детектится LLM; меню приезжает с первым
  ответом.
- «Перенести»/«Отменить» без активной записи → существующие
  `resched_none`/`cancel_none`.
- Смена языка посреди сценария → язык меняется, текущий шаг повторяется на
  новом языке (существующий механизм `_with_reprompt`). Постоянная
  reply-клавиатура при этом перерисуется со следующим menu-сообщением:
  в сообщении повтора шага живут inline-кнопки, а reply_markup в Telegram
  один — две клавиатуры в одно сообщение не помещаются.
- Telegram: `reply_markup` один на сообщение → persistent-клавиатура остаётся
  от прошлых сообщений на шагах с inline-кнопками (так Telegram и работает);
  единственный конфликт — contact-шаг, закрыт восстановлением меню.
- `escalated`-состояние: меню перехватывается, но отвечаем `escalated`-заглушкой
  как и на текст (стоп-состояние не обходится кнопками).

## Соответствие BRIEF

- P0 «дисклеймер при первом контакте» — сохраняется (приветствие после выбора
  языка либо при первом тексте).
- «Определение языка на сообщение» — сохраняется для свободного текста.
- «LLM — рот, код — мозг» — усиливается: happy path полностью детерминирован.
- Цель «70% без человека» — кнопки снижают NLU-ошибки и эскалации.

## Тесты (pytest, существующая инфраструктура)

- /start первый: экран языка → выбор → приветствие + меню.
- /start повторный: меню без выбора языка.
- Перехват меню НЕ дёргает экстрактор (fake extractor со счётчиком вызовов).
- Меню посреди booking c hold'ом → hold отменён, контекст чист, новый сценарий.
- Меню на шагах имени/телефона → прерывание без ухода PII в NLU.
- Смена языка → перерисованное меню + повтор текущего шага.
- Прайс-лист из каталога services (с ценой и без).
- Восстановление меню-клавиатуры после contact-шага (вместо remove_keyboard).
- Свободный текст при наличии меню → LLM-путь работает как раньше
  (регресс существующих 219 тестов: ноль ожидаемых падений, кроме
  тестов на remove_keyboard — обновляются осознанно).
