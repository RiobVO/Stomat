# Карта персональных данных Navbat

Инженерная карта данных для юридической оценки (персональные данные, РУз).
Документ описывает, **что фактически делает код** — без юридических выводов.
Каждое утверждение сверено с исходниками; ссылки даны как `файл:строки`
на состоянии репозитория от 07.06.2026. Система: Telegram-бот записи
пациентов стоматологических клиник, один экземпляр на клинику.

---

## 1. Какие данные собираются

| Данные | Откуда | Где хранятся | В каком виде |
|---|---|---|---|
| Имя пациента | Текстовый ввод на шаге диалога (`src/navbat/dialog/fsm.py:476-490`) | `patient.name_encrypted` | Шифртекст AES-256-GCM (`src/navbat/crypto.py:18-31`) |
| Телефон пациента | **Только** кнопка Telegram «Поделиться контактом» (`request_contact`); ручной ввод текстом не принимается (`src/navbat/dialog/fsm.py:496-510`) | `patient.contact_hash` + `patient.phone_encrypted` | SHA-256-хэш с солью клиники — ключ поиска/дедупа (узбекский номер нормализуется к `998XXXXXXXXX`, номер любой другой страны — как есть, `src/navbat/dialog/patients.py:28-43`); рядом — шифртекст AES-256-GCM того же нормализованного номера (пересмотр 11.06: номер нужен клинике в событии календаря, миграция 0018) |
| Telegram chat_id | Метаданные входящего сообщения | `patient`, `appointment`, `conversation`, `message_queue` (миграции 0001–0003) | Открытое число (bigint) |
| Тексты сообщений пациента | Входящие Telegram-апдейты | `message_queue.payload` (jsonb) — апдейт сохраняется целиком, без фильтрации полей (`src/navbat/telegram/queue.py:38-51`) | Открытый текст; см. разд. 7 о сроках |
| Состояние диалога (язык, услуга, дата, выбранный слот) | Производные от диалога | `conversation.context` (jsonb) | Открытый JSON |
| Запись на приём (услуга, дата/время, статус) | Подтверждение диалога | `appointment` (service_id, time_range, status) | Открытые операционные данные |
| Имя врача (персонал клиники) | Онбординг клиники | `doctor.name_encrypted` | Шифртекст AES-256-GCM |

Принимается только **собственный** контакт отправителя — код сверяет
`contact.user_id == from.id` (`src/navbat/telegram/worker.py:108-111`);
чужой контакт не сохраняется. Принимается номер любой страны (кнопка
отдаёт подлинный номер аккаунта); нераспознаваемый номер (не 7–15 цифр)
не сохраняется ни в каком виде — бот повторяет кнопку контакта.

### Данные о здоровье

Структурированные медицинские данные **не хранятся**: нет полей диагнозов,
жалоб, анамнеза. NLU-модель возвращает булев флаг `is_medical`
(сообщение похоже на медицинский вопрос) — он используется один раз для
показа дисклеймера «бот не даёт медицинских советов» и в БД не пишется
(`src/navbat/dialog/fsm.py:280-286`). **Однако**: свободный текст пациента
может содержать описание симптомов («болит зуб», «опухла десна») — такой
текст сохраняется в `message_queue.payload` как часть сырого сообщения
и уходит в LLM (разд. 4).

### Что НЕ собирается

Полей нет ни в одной таблице БД (проверено по всем миграциям 0001–0008):
email, почтовый адрес, дата рождения, паспортные данные, ПИНФЛ/ИНН,
платёжные/карточные данные, фото, геолокация. Telegram username
в выделенные колонки не сохраняется (присутствует только внутри сырого
апдейта в `message_queue.payload`, если Telegram его прислал).

## 2. Как данные хранятся и защищаются

- **БД**: PostgreSQL, отдельная на инсталляцию (docker-compose).
- **Имя пациента**: шифрование на стороне приложения, AES-256-GCM,
  ключ — 32 байта из переменной окружения `NAVBAT_ENC_KEY`; без ключа
  приложение падает, тихого плейнтекста нет (`src/navbat/crypto.py:18-31`).
  Тем же механизмом шифруются токен Telegram-бота и OAuth refresh-токен
  Google.
- **Телефон**: в таблице `patient` — SHA-256-хэш с per-clinic солью
  (ключ поиска) и AES-256-GCM-шифртекст номера (`phone_encrypted`,
  миграция 0018; пересмотр 11.06 — номер нужен клинике в календаре).
  Колонки с **открытым** номером в схеме по-прежнему нет
  (`src/navbat/dialog/patients.py`). Транзит — разд. 7.
- **Изоляция клиник**: PostgreSQL Row-Level Security на всех
  tenant-таблицах (`ENABLE` + `FORCE`, политика по
  `clinic_id = current_setting('app.clinic_id')`); приложение работает
  под непривилегированной ролью `navbat_app`, тенант-контекст ставится
  `SET LOCAL` на транзакцию (`src/navbat/db/base.py:28-43`, миграция 0001).
- **Секреты**: локальный `.env` (в `.gitignore`), в репозитории
  отсутствуют.
- **Аудит** (`appointment_audit`): только статусы и время приёмов
  (hold/confirm/cancel/reschedule), без имён/телефонов/chat_id
  (`src/navbat/scheduling/engine.py:319-333`).
- **Учёт LLM** (`llm_usage`): только числовые счётчики (запросы, токены,
  сбои) — текстов сообщений нет (миграции 0006–0007).
- **Логи приложения**: тексты сообщений, имена и телефоны в logger-вызовы
  не передаются; логируются служебные id апдейтов и тексты исключений
  (`src/navbat/telegram/worker.py:70,92,158`). Исключение — разд. 8.

## 4. Передача в LLM (OpenAI / Gemini) — детально

Критичный для оценки блок: текст пациента уходит провайдерам за пределами
РУз.

**Что уходит.** Ровно **одно** текущее сообщение пациента + системный
промпт (инструкции по разметке, без персональных данных,
`src/navbat/nlu/prompts/system.md`). История диалога, слоты, chat_id,
имя клиники, имена врачей в запрос **не** включаются — NLU stateless
(`src/navbat/nlu/extractor.py:3-9`).

**Что возвращается.** Строгий JSON из 7 полей: intent, service, doctor,
date_ref, time_ref, language, is_medical (`src/navbat/nlu/schema.py:26-47`).
Ответ LLM в БД не сохраняется; в БД оседают только производные
(выбранная услуга, дата, язык).

**Деидентификация перед отправкой** (`src/navbat/nlu/wrappers.py:44-51`):
телефоноподобные последовательности (7+ цифр) заменяются на литерал
`[phone]` регулярным выражением. **Маскируется только телефон** —
отдельного вырезания имён из свободного текста нет.

**Что в LLM не уходит вообще** (ноль вызовов):
- нажатия кнопок: /start, выбор языка, пункты меню, выбор слота/даты/услуги,
  кнопки напоминаний — перехватываются кодом до NLU
  (`src/navbat/dialog/fsm.py:137-152`);
- телефон, переданный кнопкой контакта, — отдельный путь обработки,
  минующий NLU (`src/navbat/dialog/fsm.py:512-546`);
- текст на шагах ввода имени и телефона — **кроме вопросоподобного**
  (см. ниже).

**Исключение, важное для оценки** (`src/navbat/dialog/fsm.py:484-489`):
если на шаге ввода имени пациент пишет вопросоподобный текст (содержит
«?» или длиннее 30 символов — например, «Меня зовут Гульнора, а сколько
стоит чистка?»), это сообщение целиком уходит в LLM, чтобы ответить
на вопрос. Телефоны в нём маскируются, имя — нет. Аналогично: имя,
упомянутое пациентом в свободной фразе на любом шаге («запишите Азиза
на завтра»), уходит в LLM как есть. Деидентификация гарантирует
отсутствие в LLM телефонов, но не имён в свободном тексте.

**Условия и ограничения вызовов:**
- fallback на Gemini — только при недоступности OpenAI (сеть/5xx/429),
  не при некорректном ответе (`src/navbat/nlu/fallback.py:23-28`);
  включается наличием `GEMINI_API_KEY`;
- дневной лимит токенов на клинику (`NAVBAT_DAILY_TOKEN_CAP`,
  по умолчанию 200 000) — при исчерпании вызов не делается
  (`src/navbat/nlu/wrappers.py:170-181`);
- rate-limit 5 сообщений / 10 с на чат — до NLU не доходит
  (`src/navbat/telegram/worker.py:41-42`);
- таймаут запроса 8 с, у OpenAI до 2 SDK-ретраев (повторная отправка
  того же текста при сетевом сбое);
- без флага `--real` внешние LLM не вызываются вовсе (локальный
  фикстурный экстрактор) — режим разработки и демо.

Договорные аспекты провайдеров (retention на стороне OpenAI/Google,
регион обработки, DPA) кодом не регулируются — это конфигурация
аккаунтов, вне репозитория.

## 5. Google Calendar

В календарь врача (аккаунт Google клиники) экспортируется событие:
`summary = "<услуга по-русски> — <имя пациента> (Navbat)"`, описание
`Телефон: <номер>` (расшифрованные `name_encrypted`/`phone_encrypted`),
время начала/конца, внутренний UUID записи в `extendedProperties.private`
(`src/navbat/calendar/sync.py:109-127`). **Имя и телефон пациента уходят
в Google-аккаунт клиники — осознанное решение владельца продукта
(11.06.2026): владелец работает из календаря, событие должно отвечать
«кто и зачем».** Отсутствующие поля (запись без пациента, NULL-колонки)
пропускаются; chat_id в событие не записывается.

При импорте ручных событий из календаря в БД бота сохраняются только
id события и временной интервал; текст/описание чужих событий
не затягиваются (`src/navbat/calendar/sync.py:241-252`).

OAuth-токен доступа к календарю хранится зашифрованным (AES-256-GCM,
`clinic.gcal_refresh_token_encrypted`). Scope — полный доступ
к календарю (`https://www.googleapis.com/auth/calendar`).

## 6. Права субъекта данных

**Анонимизация — команда `/forget <chat_id>`** в админ-чате клиники
(`src/navbat/telegram/worker.py:191-227`). Атомарно, в одной транзакции:

| Действие | Данные |
|---|---|
| Стирается (NULL) | `patient.name_encrypted`, `patient.contact_hash`, `patient.tg_chat_id` |
| Обезличивается | `appointment.tg_chat_id → NULL` — история приёмов остаётся, но отвязывается от человека |
| Удаляется физически (DELETE) | Весь диалог (`conversation`) и все сырые сообщения (`message_queue`) данного чата, в любом статусе |
| Гасится | Pending-напоминания (статус `cancelled`) — чтобы не писать в «забытый» чат |

Будущие записи при `/forget` **не отменяются** — запрос на удаление
данных не трактуется как отмена приёма (админу предлагается отменить
отдельно, если пациент просил). Подтверждено тестом
`tests/test_tg_forget.py`.

Что `/forget` **не** затрагивает (инженерные факты для оценки):
- событие в Google Calendar остаётся, включая имя и телефон в теле
  события (выгружены до `/forget`; бот тело НЕ перезаписывает — при
  переносе patch-путь обновляет только время, полная пересборка тела
  происходит лишь при ручном удалении события) — при необходимости
  клиника удаляет событие в календаре вручную;
- `appointment.tg_message_id` (id сообщения-подтверждения) не обнуляется —
  без chat_id человека не идентифицирует;
- ранее отправленные эскалационные сообщения в Telegram-чате
  администратора (разд. 8) — бот не может удалить историю чужого чата.

**Инициатор**: только администратор клиники из админ-чата. Команды
самостоятельного удаления у пациента нет; механизма выгрузки данных
по запросу субъекта (right to access / portability) в коде нет.

**Согласие**: приветственное сообщение первого контакта представляет
бота и предупреждает, что по медицинским вопросам ответит врач
(`src/navbat/dialog/replies.py:188-195`). Текста о согласии на обработку
персональных данных, политики конфиденциальности или ссылки на неё
в строках бота **нет** (проверено поиском по всем репликам).

## 7. Сроки хранения (retention)

Автоматическая чистка раз в календарный день (`src/navbat/retention.py`,
запускается из фонового цикла напоминаний,
`src/navbat/reminders.py:193-209`). Порог — `NAVBAT_RETENTION_DAYS`,
по умолчанию **90 дней**:

| Данные | Срок | Способ |
|---|---|---|
| Сырые сообщения (`message_queue`, статусы done/failed) | 90 дней | Физический DELETE |
| Неактивные диалоги (`conversation`, включая брошенные и эскалированные) | 90 дней с последнего обновления | Физический DELETE |
| Записи на приём (`appointment`) | Бессрочно | Операционная история клиники, не чистится |
| Аудит статусов (`appointment_audit`) | Бессрочно | Без персональных данных (только статусы/время) |
| Счётчики LLM (`llm_usage`) | Бессрочно | Только числа, без персональных данных |
| Данные пациента (`patient`) | До `/forget` | Имя/телефон-хэш живут, пока пациент не «забыт» |

**Транзитное хранение открытых данных, важное для оценки.** Сырой
Telegram-апдейт сохраняется в `message_queue.payload` целиком. Это
означает, что до retention-чистки (≤90 дней) или `/forget` в БД в
открытом виде лежат: тексты сообщений пациента (включая возможные
упоминания симптомов и имя, если пациент его написал) и Telegram-метаданные
(`from.id`, `first_name`, `username`). **Номер телефона из кнопки контакта
в очередь открытым НЕ сохраняется**: на границе enqueue
(`src/navbat/telegram/queue.py`, `_redact_contact_phone`) он хэшируется
и шифруется AES-256-GCM, и в `message_queue.payload` вместо
`phone_number` лежат `message.contact.phone_hash` (тот же SHA-256-хэш
с солью, что и в таблице `patient`) и `message.contact.phone_encrypted`
(шифртекст для `patient.phone_encrypted`, пересмотр 11.06) — открытого
номера нет ни в одной строке очереди.
Аналогично, имя пациента
между шагом ввода имени и подтверждением записи лежит открытым
в `conversation.context['pending_name']` и вычищается из контекста
после оформления (`src/navbat/dialog/fsm.py:753-755`); брошенный диалог
с именем в контексте живёт до retention.

## 8. Экспозиция данных персоналу клиники

Администратор клиники видит в своём Telegram-чате:

- **Эскалации** (бот передаёт диалог человеку): chat_id пациента,
  статичная причина и текущий контекст диалога в JSON
  (`src/navbat/telegram/escalation.py:26-31`). Если эскалация происходит
  между вводом имени и подтверждением записи, контекст содержит
  **имя пациента открытым текстом** (`pending_name`). Телефона в контексте
  не бывает (он хэшируется сразу и в контекст не пишется). Текст самого
  сообщения пациента в алерт не подставляется — причины эскалации
  захардкожены.
- **/stats и вечерний дайджест**: только числовые агрегаты (записи,
  отмены, эскалации, напоминания, токены, предотвращённые неявки) —
  без имён/телефонов (`src/navbat/stats.py:19-30,99-109`).
- **Алерты NLU-дрифта и бюджета**: только счётчики.

Пациент видит в подтверждениях/напоминаниях только услугу, время и имя
врача — свои данные ему не возвращаются.

## 9. Известные инженерные ограничения (честный список)

Факты, которые юристу стоит учесть при оценке; код ведёт себя так
осознанно либо ограничение признано и зафиксировано:

1. **Имя в свободном тексте уходит в LLM.** Деидентификация вырезает
   только телефоны; имя, написанное пациентом в свободной фразе или
   в вопросоподобном тексте на шаге имени, передаётся в OpenAI/Gemini
   (разд. 4).
2. **Телефон хранится шифрованным, не только хэшем** (пересмотр 11.06,
   решение владельца продукта: номер нужен клинике в событии календаря).
   Номер из кнопки контакта на границе enqueue хэшируется И шифруется
   AES-256-GCM; открытым он не лежит ни в очереди, ни в `patient`
   (разд. 2, 7) — но при компрометации `NAVBAT_ENC_KEY` вместе с БД
   номер восстановим, как и имя. `/forget` обнуляет шифртекст вместе
   с хэшем и именем.
3. **Хэш телефона — псевдонимизация, не анонимизация.** Соль хранится
   в той же БД открытым текстом (`clinic.salt`); пространство узбекских
   номеров ~10⁹ — при компрометации БД хэш обратим перебором.
   Для демо-клиники соль захардкожена (`src/navbat/onboard.py:63-64`);
   генерации криптостойкой соли при онбординге боевой клиники в коде нет.
4. **Имя в админ-чате при эскалации.** Контекст диалога с `pending_name`
   уходит сообщением в Telegram-чат администратора (разд. 8); `/forget`
   эту историю не очищает.
5. **Шифрование на уровне приложения покрывает только имена и токены.**
   chat_id, тексты в `message_queue`/`conversation` лежат в БД открытыми;
   шифрование диска/БД — вопрос инфраструктуры, не кода.
6. **Нет механизма согласия.** Бот не запрашивает и не фиксирует согласие
   на обработку данных; текстов о ПД в репликах нет (разд. 6).
7. **Нет выгрузки данных субъекту** — только анонимизация через
   администратора.
8. **Ротация ключа шифрования не реализована** — один активный
   `NAVBAT_ENC_KEY`, перешифровки существующих данных при смене нет.
9. **Бэкапы в коде не реализованы** — политика бэкапов (и их шифрование)
   будет определяться на этапе развёртывания, в репозитории отсутствует.
