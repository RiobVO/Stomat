# Упрочнение до пилота (pilot-hardening)

Чеклист дыр из состязательного аудита кода 08.06.2026 (3 независимых
прохода: архитектура, баги, тесты). Цель — «инженерная готовность к
пилоту = 100%»: закрыть всё, что позорит на живых пациентах или течёт
PII, затем отрефакторить ядро. Каждая дыра верифицирована в коде вручную
до внесения в список. Фиксы — строго TDD (RED воспроизводит → GREEN).

**Вне рамки этого чеклиста** (решения пользователя): группа C
(деплой/эксплуатация), NLU-валидация на живом потоке и платные прогоны,
полное видение продукта из BRIEF (голос, WhatsApp, МИС, веб-дашборд).

Оценка аудита на старте: ядро кода 62%, готовность к пациентам 70%,
ценность тестов 68%.

## P0 — безопасность данных (блокеры пилота)

- [ ] **C1. `--real` шлёт PII в OpenAI без деидентификации.**
  `telegram/app.py:65-78`: `build_extractor(use_real=True)` возвращает
  голый `OpenAIExtractor()` — без `DeidentifyingExtractor`, без token-cap,
  без drift. Путь `python -m navbat.telegram --clinic <uuid> --real`.
  Свободный текст с телефоном уходит в OpenAI открытым. Прод-путь
  (`supervisor.build_real_extractor`) маскирует — расхождение.
  Противоречит docs/PRIVACY.md. Фикс: собрать ту же цепочку обёрток.

- [ ] **M2. Длинные имена уходят в LLM.** `fsm.py:53-54,485-486`:
  `_looks_like_question` = `"?" in message or len > 30`. На шаге
  `awaiting_name` имя длиннее 30 символов (узбекское ФИО с отчеством —
  «Абдурахмонов Жасурбек Шухратович» = 31) считается вопросом и уходит
  в `_try_extract` → OpenAI. Вторая PII-утечка. Фикс: на шаге имени НЕ
  гнать в LLM по длине — только по явному «?» (или убрать длину для
  PII-шагов).

## P1 — ложные эскалации (портят happy-path и метрику «бот сам»)

- [ ] **M1. Некруглое время → «нет слотов» + эскалация.**
  `dates.py:51-61` + `schema.py:18` (`TIME_REF_RE` допускает любое
  `HH:MM`) + `fsm.py:338-342`. Пациент «на 14:15», сетка слотов 30 мин →
  точного совпадения нет → `no_slots_at_all` + эскалация «нет слотов на
  2 недели». Самая частая дыра (люди называют некруглое время). Фикс:
  точное время = предпочтение, не жёсткий фильтр — слот, ПОКРЫВАЮЩИЙ
  время, или ближайшие слоты дня вместо капитуляции.

- [ ] **C2. Блокировка/удаление бота → шторм ретраев + ложные эскалации.**
  `worker.py:62-82`, `api.py:114-119`. «bot was blocked»/«chat not found»
  → `TelegramAPIError` → `fail()` → 3 ретрая → dead-letter + эскалация
  админу. Каждый заблокировавший пациент спамит админ-чат + сериализация
  чата (`queue.py`) держит немоту 15+ мин. Фикс: логические 4xx Telegram
  (403/400 chat not found) — не ретраить и не эскалировать, тихо гасить.

- [ ] **m2. book↔question шире, чем прикрыто.** `fsm.py:243-259,726`.
  Бэкстоп ловит `question` только при наличии `date_ref`/`time_ref`.
  «У вас есть свободное время?» без даты → эскалация «вопрос вне
  компетенции» вместо показа слотов. Фикс: вопрос о наличии без даты —
  трактовать как book (показать ближайшие слоты).

## P2 — minor

- [ ] **m1. Имя в эскалационном алерте.** `fsm.py:527-528`,
  `escalation.py:28`. При не-узбекском номере `conv.context` (с
  `pending_name`) уходит в админ-чат и логи. Имя пациента утекает
  персоналу. Фикс: чистить PII-ключи из context перед notify, либо слать
  whitelisted-поля.

- [ ] **m3. `ctx["service"]` без защиты в callback.** `fsm.py:460`.
  При потерянном контексте (рестарт + устаревший `tg_actions`, гонка
  M4) → `KeyError` → fail → ретраи → эскалация. Фикс: graceful при
  отсутствии ключа — повтор шага/«кнопка устарела».

- [ ] **m4. `evening` = до конца суток.** `dates.py:18`:
  `evening=(17:00, time.max)` — слот в 23:30 формально «вечер». Сейчас
  безвредно (рабочее окно отрежет), мина при суточных графиках. Фикс:
  ограничить разумным концом (напр. 21:00).

## P3 — рефакторинг ядра (62% → ~85% зрелости)

Фундамент (БД, scheduling-движок, тесты, обработка ошибок, RLS, PII-
границы) — честно крепкий, переписывать с нуля НЕ нужно. Слабое звено —
диалоговый слой.

- [ ] **R1. Слой данных вместо сырого SQL в FSM.** 144 инлайновых
  `text("SELECT...")` в 24 файлах, 18 в `fsm.py`. Паттерн репозиториев
  в проекте УЖЕ есть (`patients.py`, `conversation.py`) — применить к FSM.
- [ ] **R2. Типизированное состояние диалога.** `conv.context: dict` со
  свалкой строковых ключей + ручной `_BOOKING_KEYS`. Заменить на
  dataclass — забытый ключ ловит тип-чекер, а не прод.
- [ ] **R3. Единый каталог услуг.** 9 услуг продублированы в 4 местах
  (`schema.py`, `gemini_extractor.py`, `replies.py`, `onboard.py`).
  Свести к одному источнику.
- [ ] **R4. Разбить god-object `DialogEngine`** (849 строк, ~10
  ответственностей) по сценариям (booking/reschedule/cancel) за общим
  роутером.

## P4 — слепые зоны тестов (что отметить, не обязательно код)

- [ ] **T1. NLU-качество не защищено регрессией.** Главная фича
  («понимает узбекский») — ни одного теста; смена промпта/модели/провайдера
  деградирует узбекский, CI зелёный. Требует офлайн-eval в процессе (не
  обязательно CI — упирается в деньги/решение пользователя).
- [ ] **T2. Gemini-failover на живом ответе не проверен** (решение:
  free tier, billing не включаем — риск осознан).
