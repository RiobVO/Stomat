# Navbat — продуктово-инженерный бриф (v5)

> v4 — финал ревью (аудиты, заморозка). v5 — смена стратегии: **строим до
> инженерных 100%, потом пилот, потом продажи**. Никаких «для демо сойдёт»:
> продукт перед первым касанием клиента — в production-качестве, без костылей.
> Пилот — не продажа недоделки, а контролируемый запуск готового продукта на
> одной клинике: он закрывает то, что физически нельзя закрыть без живых
> пациентов (NLU на реальном потоке, узбекские тексты, процессы клиник).
> Деньги берём только после пилота. Скоуп «100%» = чеклист
> «Production-Ready v1.0» (раздел 14), не бесконечная полировка.

**AI-администратор в Telegram, записывает пациентов частной стоматологии 24/7 на узбекском и русском — без накладок, без потерянных вечерних заявок.**
**Модель: разовая настройка + ежемесячное обслуживание (хостинг + токены + поддержка + SLA). Хостинг на стороне разработчика, multi-tenant.**

---

## Принятые решения (развилки)

1. **Коммерция:** разовая настройка (3–5 млн) + ежемесячное обслуживание (~300–500к). НЕ «купил + сюрприз-счёт за токены». Рефрейм: «мы держим и обслуживаем твоего администратора» — сервис, не подписка на софт.
2. **Tenancy:** multi-tenant, единый хостинг разработчика, одна БД + RLS. Код multi-tenant с дня один; флот-операционка фазируется по росту клиник.
3. **Голос:** text-only в v1, голос (Whisper) в P1. УЗ-рынок без голоса теряет конверсию, но ядро валидируем на тексте.
4. **Модель:** самый дешёвый GPT на старте/в спайке (это и есть тест гипотезы №1). Прод-модель — после теста NLU + юр-ответа по трансграничной передаче. За адаптером, свапается в одном месте.
5. **UX: кнопки-first, LLM-fallback.** /start → выбор языка → постоянное reply-меню (Записаться/Перенести/Отменить/Цены/Язык); нажатия перехватываются до NLU — ноль токенов на happy path. Свободный текст на любом шаге понимает LLM (узб/рус/суржик) и ведёт по тому же FSM — дифференциатор продукта сохранён, LLM работает только там, где даёт ценность.
6. **Стратегия запуска (v5):** инженерные 100% (чеклист Production-Ready v1.0) → бесплатный пилот на 1 клинике как запуск готового продукта → продажи. Не «демо → продажа → доделка».

## 1. Целевой клиент (beachhead)

Частные стоматологии Ташкента на 1–4 кресла, owner-operated, поток заявок из Instagram/Telegram.

Почему они: высокий чек (одна спасённая запись > стоимости настройки), острая считаемая боль от неявок и потерянных вечерних заявок, владелец решает сам (короткий цикл), их много в одном городе.

Преимущество над Yclients/DIKIDI/Medesk: бот в Telegram, где пациент уже находится — нулевое трение.

Салоны, барбершопы, клиники общего профиля — после 10 продаж в стоматологиях.

---

## 2. Проблема

- Половина заявок падает вечером/в выходные → пациент уходит к конкуренту.
- Неявки выжигают кресло-часы; напоминания делаются руками и забываются.
- Время админа — на рутину: цена, адрес, часы, переносы.

---

## 3. Что делает

Два входа в один сценарий: **кнопки** (/start → язык → постоянное меню → услуга → день → слот, ноль LLM-токенов) и **свободный текст** (узб/рус, латиница/кириллица/суржик — LLM вытаскивает намерение и слоты одной фразой, пропуская кнопочные шаги). Дальше одинаково: реальные свободные слоты → запись / перенос / отмена → подтверждение → напоминания за 24 ч и 2 ч → при сложном/непонятном передаёт человеку с контекстом.

**Цель: ~70% диалогов без человека.** Бот не заменяет админа, убирает рутину.

## 4. Объём v1 и не-цели

### P0 — без этого не продаётся
- Telegram; узб + рус (латиница + кириллица + суржик).
- Извлечение намерения/сущностей → валидный JSON по схеме (с проверкой и repair).
- Реальные свободные слоты; запись / перенос / отмена.
- Напоминания 24 ч / 2 ч, вычисляются из БД, с трекингом доставки.
- Эскалация с контекстом: сложный вопрос, жалоба, медвопрос, 2 неудачных уточнения.
- Дисклеймер при первом контакте: «Я виртуальный администратор клиники X, помогу с записью. По медицине — к врачу.»
- Поведение в нерабочее время: принимает, говорит «закрыто», предлагает слоты следующего рабочего дня.
- Закрытые дни и выходные: клиника сама закрывает/открывает день командой /dayoff (/dayopen) в админ-чате — Навруз/Хайиты по желанию (многие частные работают в праздники); Рамадан двигает часы.
- Уведомление эскалаций в Telegram-чат админа.
- Дашборд: записи / предотвращённые неявки / сохранённая выручка в сумах.

### P1 — быстрый follow-up
- Голос (Whisper-транскрибация) — для УЗ это near-must.
- WhatsApp как второй канал.
- Список ожидания: слот освободился → предложение желающим.
- Опросник после приёма.
- Команда админу: быстрая блокировка времени прямо в Telegram.

### P2 — архитектурная страховка
- Instagram Direct.
- Прямая интеграция с МИС (Dentafill, IDENT, DIKIDI) — только при вменяемом API.
- Онлайн-оплата.

### Не-цели
- Никаких медицинских ответов ботом.
- Никакой CRM и маркетинговых рассылок.
- В v1 голоса нет (но он в P1, не «никогда»).

## 5. Архитектура

### Принцип
**LLM — рот, код — мозг.** Модель достаёт намерение и генерирует текст. Все решения о слотах, правилах, переходах — детерминированный код.

### Tenancy
Multi-tenant: единый хостинг, одна PostgreSQL с `clinic_id` во всех таблицах + **Row-Level Security** для изоляции арендаторов. Каждая клиника = свой Telegram-бот-токен и свой Google-аккаунт, но всё в одном кластере. Флот-операционка (центральный мониторинг, веерные миграции, canary-раскатка конфига) — добавляется по мере роста, не на нулевой клинике.

### Компоненты
- **Channel adapter** — Telegram, webhook + HTTPS. `restart: unless-stopped`. **Проверка валидности cert встроена в health-check** (cert протух → TG молча перестаёт слать; ловим заранее).
- **Durable message queue** — Postgres-backed (`FOR UPDATE SKIP LOCKED`), без Redis на этом масштабе. Вебхук подтверждается мгновенно (200), обработка в очереди. **Ack только после успешной обработки** (иначе при падении обработчика сообщение потеряно — TG не повторит).
- **Сериализация per-chat (FSM concurrency — был баг):** пациент шлёт «Завтра», «нет, послезавтра», «в 18:00» за 2 сек → три вебхука почти одновременно → три воркера читают один стейт `conversation` и пишут поверх друг друга → FSM рвётся. Фикс: пессимистичная блокировка на чат — `pg_advisory_xact_lock(hashtext(chat_id))` или `SELECT ... FOR UPDATE` строки `conversation` на время обработки апдейта; остальные сообщения чата ждут. Обработка **в порядке `update_id`** (Telegram-монотонный): блокировка решает параллельность, порядок решает update_id.
- **Orchestrator / FSM** — конечный автомат, переходы = код. Поддержка **прерываний и возврата состояния**: вопрос вбок («а где вы?») посреди записи → ответ + возврат в прежнее состояние, не сброс.
- **NLU (LLM)** — slot filling → JSON по схеме с валидацией и retry-repair при кривом JSON. **LLM извлекает относительную ссылку времени («завтра», «после обеда», «ertaga»), а абсолютную дату/время в TZ клиники считает детерминированный код** — модели нельзя доверять арифметику дат. Перед отправкой — **деидентификация** (имя/телефон вырезаются). **Защита от prompt-injection**: бот не выполняет инструкции из текста пациента вне своей роли и не раскрывает системный промпт. Промпт версионируется в БД.
- **Управление контекстом (решено в Ф1: stateless-NLU):** в LLM уходит ОДНО сообщение пациента без истории — контекст (слоты) копит FSM в `conversation.context`. Риск token bloat снят by design: одно сообщение = естественный потолок запроса. Rolling window из v4 не понадобился — точность подтверждена спайком (410 фраз) и кодовым ядром Ф1. Историю в промпт не добавлять без перепрогона eval (зафиксировано в докстринге `navbat/nlu/extractor.py`).
- **Scheduling engine (детерминированный)** — единственный хозяин занятости. Правила длительности/часов/перерывов/буфера/выходных/праздников.
- **Calendar adapter** — Google Calendar. **Push-уведомления (watch API), а не поллинг** → изменения сразу, окно гонки схлопывается. Авто-refresh OAuth + алерт при провале. **Верификация Google-приложения** (иначе sensitive-scope токены дохнут за 7 дней). Conflict-resolution при пуле описан явно (см. целостность). **Квоты Google Cloud (флот-баг):** OAuth-приложение одно на весь флот → квоты Calendar API считаются на уровне ТВОЕГО проекта, не клиники. 10–15 клиник с watch + постоянными create/move/cancel упрутся в дефолтную квоту → 429 → записи падают. Фикс: запрос увеличения квот заранее + распределение по нескольким Cloud-проектам при росте + мониторинг приближения к лимиту с дня один.
- **LLM provider adapter** — модель за абстракцией. Основной провайдер + **вторичный fallback** (аутэйдж основного → переключение, иначе ночью бот эскалирует ВСЁ — когда человека нет).
- **Notifier** — напоминания вычисляются запросом к `appointment` на старте и в цикле (НЕ таймеры в памяти — иначе теряются на каждом деплое). Retry: backoff ×3 → dead letter → алерт. Статус: `sent | delivered | failed`.
- **Human handoff** — уведомление с полным контекстом в Telegram-чат админа.
- **Config service** — бот тянет промпт/прайс/расписание. **Canary + per-clinic pinning + раскатка по одной клинике** (иначе плохой промпт деградирует весь флот разом).
- **Deep health endpoint** — `/health`: `SELECT 1` + timestamp последней синхры Calendar < 5 мин + тестовый вызов LLM + валидность cert.
- **Storage** — PostgreSQL, `btree_gist` + RLS. **Один стек: Postgres делает всё** — exclusion constraints, RLS, `tsrange`, блокировки (`FOR UPDATE`, advisory locks), очередь (`SKIP LOCKED`). Redis на этом масштабе НЕ нужен: меньше движущихся частей, меньше что ломать соло-оператору. Добавить позже, если упрёшься в throughput — обратимое решение.

### Модель данных
```sql
CREATE EXTENSION IF NOT EXISTS btree_gist;

-- нормализация телефона к 998XXXXXXXXX ДО хеширования (иначе дедуп/история/вейтлист рассыпаются)

CREATE TABLE clinic (id uuid PRIMARY KEY, name text, salt text, timezone text DEFAULT 'Asia/Tashkent');

CREATE TABLE service (id uuid PRIMARY KEY, clinic_id uuid NOT NULL, name text, aliases text[], duration_min int, price numeric);
-- aliases: ['чистка','гигиена','tish tozalash'] — резолвинг услуги против синонимов

CREATE TABLE doctor (
  id uuid PRIMARY KEY, clinic_id uuid NOT NULL,
  name_encrypted text,
  working_intervals jsonb,  -- {mon:[[9,13],[14,18]], ...} массивы интервалов: обед, две смены
  buffer_min int DEFAULT 10
);

CREATE TABLE holiday (id uuid PRIMARY KEY, clinic_id uuid, date date, reason text);
-- закрытые дни: клиника сама управляет через /dayoff и /dayopen в админ-чате

CREATE TABLE patient (
  id uuid PRIMARY KEY, clinic_id uuid NOT NULL,
  contact_hash text,         -- SHA-256(normalized_phone + clinic.salt)
  name_encrypted text,
  tg_chat_id bigint          -- durable: для вейтлиста, опросника, узнавания вернувшегося
);

CREATE TABLE conversation (   -- БЫЛА ПРОПУЩЕНА в v2; core-таблица
  id uuid PRIMARY KEY, clinic_id uuid NOT NULL,
  tg_chat_id bigint, patient_id uuid,
  fsm_state text, context jsonb,   -- переживает рестарт; FSM-state и LLM-контекст живут согласованно
  updated_at timestamptz
);

CREATE TYPE appt_status AS ENUM ('hold','booked','cancelled','done');

CREATE TABLE appointment (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  clinic_id uuid NOT NULL, doctor_id uuid, service_id uuid, patient_id uuid,
  time_range tsrange,        -- УЖЕ расширен на buffer_min при вставке (буфер на уровне БД)
  status appt_status DEFAULT 'hold',
  source text,               -- bot | manual | phone | gcal_import
  lang char(2) DEFAULT 'ru',
  hold_expires_at timestamptz,
  tg_chat_id bigint, tg_message_id bigint,
  created_at timestamptz DEFAULT now(),
  UNIQUE (tg_chat_id, tg_message_id),   -- идемпотентность: ключ (chat_id, message_id), НЕ голый message_id
  EXCLUDE USING gist (doctor_id WITH =, time_range WITH &&) WHERE (status IN ('hold','booked'))
);
-- RLS на всех таблицах: policy USING (clinic_id = current_setting('app.clinic_id')::uuid)
```

### Целостность записи
- **Soft-hold:** статус `hold`, `hold_expires_at = now()+3min`. Слот считается свободным прямо в запросе если `hold_expires_at < now()` — НЕ полагаемся только на фоновый джоб (джоб подвис → слоты залочены).
- **Гонка двоих:** exclusion constraint отклоняет второго.
- **Буфер в constraint:** `time_range` расширяется на `buffer_min` при вставке → БД не даст записи встать вплотную (раньше буфер был только в коде = слабая гарантия).
- **Идемпотентность:** `UNIQUE (tg_chat_id, tg_message_id)` — message_id уникален лишь в пределах чата, голый ключ был багом.
- **Conflict-resolution с Calendar (главная дыра v2):** push-уведомления Google + при создании записи ботом — финальная перепроверка перед подтверждением. Если ручное событие из Calendar конфликтует с записью бота при пуле — приоритет у ручного (человек физически в кресле), бот переносит свою запись и уведомляет пациента + админа. Не молчаливое отклонение.
- **TZ:** хранение UTC, отображение Asia/Tashkent (UTC+5, без DST).

### Надёжность
- **Graceful degradation:** таймаут LLM 8 сек → fallback-провайдер; если и он лёг → «передаю администратору» + handoff.
- **Rate limit + per-clinic token cap:** ≤5 сообщений/10 сек на chat_id; дневной/месячный лимит токенов на клинику + circuit breaker + kill-switch + алерт «клиника #N жжёт 5× нормы». Защита от спама/DDoS на твой кошелёк.
- **Retry уведомлений:** backoff ×3 → dead letter → алерт.

### Язык и промпт
Определение языка на сообщение. Промпт содержит примеры латиницы/кириллицы/суржика, словарь местных терминов и стом-сленга, правило отвечать на языке пациента. Все фолбэк-строки на узб+рус, проверены носителем до пилота.

**Гипотеза №1 (неделя 1, BLOCKING):** дешёвый GPT понимает реальные узбекские диалоги ≥ 85%. Тест-сет включает: латиница+кириллица, code-switching в одном предложении, voice-to-text артефакты, пожилых, резолвинг услуги против синонимов. Не «зелёный на курированных 50 сообщениях».

### Безопасность и PII
- **Трансграничная передача — главный юр-риск.** Текст диалога = health-данные; отправка в US/EU-LLM = трансграничная передача спецкатегории ПД, inference физически за рубежом. **VPS в Ташкенте этого НЕ закрывает.** Действия: (а) юрист по ПД РУз до прода; (б) если трансгранично нельзя — локальная/региональная модель (адаптер позволяет сменить); (в) проверить нужна ли регистрация базы ПД в госреестре и кто оператор.
- Телефон: SHA-256(нормализованный номер + clinic.salt). ФИО: AES-256.
- В LLM — деидентифицированный текст.
- БД на локальном (ташкентском) хосте.
- RLS для изоляции арендаторов.
- Secrets в env, не в коде.
- Audit log: каждое действие с appointment (actor, before, after).
- **Data subject rights:** механизм удаления данных пациента по запросу (закон РУз).
- Retention: логи диалогов чистятся через 90 дней.

### Деплой и флот-операционка
- Всё в Docker Compose (PostgreSQL + app + nginx + certbot + очередь). `restart: unless-stopped`.
- **Миграции БД через Alembic + оркестрация** (а не SSH в каждый инстанс). Версионирование схемы.
- **Staging / тестовая клиника** — новый промпт/код тестируется ДО живых пациентов.
- CI/CD: race-condition тест (50 одновременных запросов → 1 запись) автоматизирован в пайплайне.

### Наблюдаемость (централизованная)
- Структурные логи со всех клиник в один сборщик (лёгкий Loki/Grafana), error-tracking (Sentry-класс).
- Метрики: % без человека, % эскалаций, p50/p95 latency, reminder delivery, hold expiry, **дрифт точности NLU per-clinic** (алерт на деградацию).
- Fleet cost governance: токены per-clinic в реальном времени, аномалия-алерт, глобальный kill-switch.

### Бэкап и восстановление
- **WAL-архивинг (PITR) + бэкап раз в 1–2 часа** в S3/облако (24ч RPO = катастрофа: диск умер в 23:00 → потеря всех записей дня).
- **Протестированный runbook восстановления + RTO.** pg_dump без проверенного restore — это не бэкап.

## 6. Приёмочные тесты и SLA

| Тест | Результат |
|---|---|
| 50 одновременных запросов на слот | 1 запись, 0 двойных. Автоматизирован в CI |
| 3 сообщения от одного чата за 2 сек | Последовательно (per-chat lock, порядок по update_id), FSM не рвётся |
| Ручная запись в Calendar + бот на тот же слот | Conflict-resolution: приоритет ручной, бот переносит и уведомляет |
| LLM-провайдер лёг | Переключение на fallback; оба легли → эскалация |
| 2 пустых JSON подряд | Эскалация, цикл обрывается |
| Кривой JSON от модели | Валидация + repair, не падение |
| Медвопрос | Дисклеймер + эскалация |
| Дубль webhook | Одна запись (ключ chat_id+message_id) |
| Деплой / ребут VPS | Напоминания не теряются (из БД), бот поднялся сам |
| Cert на 90-й день | Health ловит заранее, алерт |

**SLA:** uptime ≥ 99% / ответ p95 < 5 сек / доставка напоминаний ≥ 95%.

## 7. Операционные требования (условия корректной работы)

- Расписание актуально в Google Calendar.
- Любая запись (звонок, walk-in) вносится администратором в день записи.
- Изменения графика — заранее, не голосом.

Нарушение — зона ответственности клиники, фиксируется в договоре. (Митигация: занос ручной записи в один клик через команду в Telegram; на пилоте меряем частоту нарушений.)

---

## 8. Онбординг (один клиент < 1 рабочего дня)

От клиники: Telegram-токен (BotFather), Google-аккаунт, услуги с ценами/длительностью/синонимами, расписание врачей (с перерывами/сменами), контакт админа. Выходные дни клиника закрывает сама (/dayoff в админ-чате).

Делает разработчик: создаёт tenant в кластере, **импортирует существующие будущие записи из Calendar** (иначе бот в первый день предложит занятые слоты), настраивает config, прогоняет приёмочные тесты + 5 тест-диалогов, обучает админа (1 ч).

## 9. Коммерческая модель

### Что продаётся
Разовая настройка + ежемесячное обслуживание. Не «купил софт», а «мы держим и обслуживаем твоего администратора».

### Цена (гипотеза — проверить на первых 3 сделках)

| Компонент | Цена |
|---|---|
| Разовая настройка (база) | 3–5 млн сум |
| Ежемесячное обслуживание (хостинг + токены + поддержка + SLA) | ~300–500 тыс сум/мес |
| Расширенный (несколько врачей, аналитика, зеркало МИС) | настройка ×1.5–2 |
| Первые 1–2 клиента (референс) | сниженная настройка, обслуживание полное |

Нарратив через деньги клиента: «обслуживание < стоимости 1 потерянного пациента в месяц». Recurring закрывает: твою маржу на токенах, поддержку, починку апстрима, и убивает конфликт «я же купил» (всё названо заранее) + дебиторку мелких сумм (сумма осмысленная, привязана к ясной ценности).

### Cost model (на клинику в месяц, твои затраты)

| Компонент | Стоимость |
|---|---|
| Доля кластера (хостинг) | ~50–100 тыс сум |
| LLM токены (дешёвый GPT) | ~30–100 тыс сум |
| Backup/S3 | ~10–20 тыс сум |
| **Итого затрат** | **~90–220 тыс сум** → маржа в обслуживании 300–500к |

Биллинг токенов: карта в USD у разработчика; per-clinic cap + circuit breaker защищают от перерасхода. Налоговый статус (самозанятый) — оформить.

### Поддержка
- Баги по вине разработчика — входят в обслуживание (не «30 дней и пока»).
- Новые фичи/доработки сверх объёма — отдельной разовой ценой.
- Апстрим-поломки (Telegram/Google) — входят в обслуживание (чинишь раз на кластере для всех).

## 10. Дашборд (язык денег)

- Бот принял X заявок → записал Y пациентов.
- Предотвращено Z неявок ≈ **сохранено ~N млн сум**.
- Топ-5 вопросов.
- Эскалаций сегодня/за неделю.

## 11. Заход на рынок

- Первые 10 клиник — руками: обход + DM владельцам в Instagram.
- Заход: бесплатный пилот 2 нед → цифры → конверсия в настройку + обслуживание.
- Reference-риск (рынок болтливый): первых клиентов выбирать лояльных, качество на пилоте критично — 2 плохих отзыва убьют следующие 8.

---

## 12. Метрики

**Leading:** пилот→оплата ≥ 40% / бот закрывает ≥ 70% без человека / p95 < 5 сек / reminder delivery ≥ 95%.

**Lagging:** 10 клиник на обслуживании за 6 мес / снижение неявок −30% за 4 нед / % по рекомендации / MRR обслуживания.

## 13. Kill criteria

| Условие | Действие |
|---|---|
| Гипотеза №1: NLU < 85% на реальном тест-сете дешёвым GPT | Модель дороже / локальная, либо пересмотр юнит-экономики. Если и сильная не тянет узбекский — стоп |
| Нет 3 согласий на пилот за 3 нед | Стоп / смена сегмента |
| Из 3 пилотов 0 оплат | Пересмотр оффера/цены |
| Бот закрывает < 40% на пилоте | Переработка NLU/FSM |
| Юрист: трансгранично нельзя и локальной модели для узбекского нет | Архитектурный/правовой блок — переосмыслить |
